# Decision Log — Ma Ville Agenda PRD

## 2026-05-29

### Activation & Intent
- **Intent detected**: Create PRD, MVP scope only
- **Working mode**: Fast path (draft full PRD with [HYPOTHÈSE] tags, user reviews and iterates)
- **Language**: French (communication + document output)
- **Scope**: MVP only. V2 (backoffice, moderation, push) and V3+ (white-label) are out of PRD scope for now.

### Scope Decisions

#### MVP Features (In)
- **F1: Liste des événements** — display all upcoming events, sorted by date; 7 real Léognan events (May 2026) as seed data
- **F2: Filtrage par catégorie** — toggle between Culture, Sport, Animation, Commerce, Autre
- **F3: Fiche détail** — tap an event to see full details (title, image, description, start/end datetime or all-day, category, optional link, location)
- **F4: Navigation par date** — browse events by week/day (mechanism TBD in architecture phase)

#### Backoffice & Validation (In MVP)
- **F5: Backoffice super-admin (web)** — login, create events, validate/reject submissions, archive/unpublish events
- **Validation workflow** : created/submitted events → "En attente" → super-admin approves → "Publié" (visible in app immediately)
- **Moderation (manual)** : Camil as sole validator in MVP; architecture ready for modos + RBAC in V2

**Rationale**: MVP is operationally functional, not just a read-only demo. Super-admin can create and manage real events. This validates the submission/validation loop and proves the flow works before scaling to multi-user (V2).

#### MVP Features (Out)
- **RBAC (multi-user)** — commerçants, associations, modos accounts → V2
- **User submissions (mobile)** — users submit events from app → V2
- Push notifications → V2
- Facebook / Instagram auto-publishing → V2
- Favorites / bookmarking
- Search by keyword
- Dark mode
- Event editing after publication (archive/unpublish only)
- Geolocation / maps

**Rationale**: These defer to V2 (which opens the platform). MVP focus: prove the core validation loop works with super-admin as sole gatekeeper.

### Platform & Technical Assumptions

| Area | Decision | Rationale |
|---|---|---|
| **Platform** | iOS + Android via cross-platform framework | Brief specifies both; cross-platform keeps dev effort solo-manageable |
| **Framework** | React Native or Flutter [TBD architecture] | Needed in architecture phase; both support Vercel/serverless backends |
| **Backend** | Managed DB or serverless API [TBD] | Zero budget → free tier (Firebase, Supabase, etc.) or personal hosting |
| **Distribution** | App Stores (iOS + Google Play) | Required for user acquisition goal |
| **API** | REST or GraphQL [TBD] | Backends mobile app + admin backoffice |
| **Admin backoffice** | Web (responsive desktop-first) | React/Next.js/Vue [TBD]; Vercel or personal hosting |
| **Database** | Managed service (Firebase, Supabase, PostgreSQL) | Free tier with zero budget; must support event CRUD + user auth |
| **Admin auth** | Password or magiclink [TBD] | Super-admin login, minimal security (no OAuth yet) |
| **Data sync** | Real-time or near-real-time [TBD] | Validated events appear in app within seconds of approval |

**Rationale**: Stack choices driven by three constraints: (1) solo dev (Camil), (2) zero budget, (3) Vercel-compatible. Backoffice adds API + web layer but framework selection (RN + same tech for admin) keeps maintainability lean.

### Success Metrics

| Metric | MVP Target | Notes |
|---|---|---|
| Downloads (3 months) | 500 | Proof of concept; indicates interest in Léognan population |
| Session frequency | Measure | Validates utility; no target set (baseline for iteration) |
| 7-day retention | Measure | Indicates habit formation vs. one-off trial |

**Rationale**: 500 downloads in 3 months is ambitious but achievable for a 30k-person commune with 5-person civic team promoting it. Primary success signal is adoption; metrics like engagement and retention are learning signals for V2 roadmap.

### Non-Goals (Explicit Out)

- **No user accounts in MVP** — read-only access reduces complexity and friction
- **No search** — filtering by category is sufficient for MVP
- **No offline support** — internet access assumed (target: smartphone users)
- **No white-label in MVP** — V3+ concern; MVP is Léognan-only

**Rationale**: Each of these can be added in V2 without rearchitecting. Deferring them keeps MVP scope achievable.

### Hypotheses (Tagged in PRD)

Key assumptions that will be validated in architecture / UX design phase:

1. **Navigation by date is a week/day toggle** — not a full calendar picker
2. **Location is text, not geo-aware** — no maps or proximity filtering
3. **Category can be represented as color + icon** — simple visual affordance
4. **Super-admin validation loop is sufficient** — manual approval/rejection is acceptable in MVP; scales to modos in V2
5. **Real-time event sync is fast enough** — validated events appear in app within seconds of approval (TBD in architecture)
6. **7 events + ability to create more is enough** — sufficient to demonstrate list, filter, detail, and creation flows
7. **Mobile app users don't submit** — submission only via backoffice in MVP; opens in V2
8. **Backoffice on web, not mobile** — super-admin uses desktop/laptop
9. **Target is phone, not tablet** — design app for portrait 360px width

**Rationale**: MVP now includes backoffice + API, making it more complex. Hypotheses will be validated during architecture and UX design phases.

### Phase Gates

- **Before development**: Architecture phase must resolve framework, backend, data layer
- **Before launch**: Internal testing with civic team; beta feedback cycle if time permits
- **Go/no-go**: Team decision + App Store review (iOS) and Google Play review (Android)

## 2026-05-29 (Updated) — Scope Adjustment: Backoffice MVP

### Context
After initial PRD draft review, Camil confirmed: "Pour le premier livrable il va quand même falloir un backoffice minimal" — the MVP must include a basic super-admin backoffice. Initial fast-path draft had treated backoffice as V2, but operational reality requires it in MVP.

### Decision
**Backoffice MVP is now IN scope.**

- **F5: Backoffice super-admin (web)** added with 5 sub-features:
  - F5a: Super-admin authentication (login/password or magiclink)
  - F5b: Create event form (all 7 fields + auto-save)
  - F5c: Validation queue (list of pending submissions)
  - F5d: Validation detail view (approve/reject with optional comments)
  - F5e: Published event history (list, archive, unpublish)

### Impacts
- **Architecture**: API + web backoffice now required (no longer mobile-only); adds REST/GraphQL layer
- **Timeline**: shifts from 4-6 weeks to ~8-12 weeks solo
- **RBAC**: deferred to V2; super-admin (Camil) is sole user in MVP
- **User submissions**: deferred to V2; only super-admin can submit in MVP

### Rationale
MVP is no longer a read-only demo. It's an operationally functional app: users browse via mobile, super-admin manages via web backoffice, validation loop proves the concept works. This sets up V2 (multi-user submissions) on solid foundation.
